## 1. 요구사항 구체화

### 1.1 문제 정의

이 서비스는 물류 담당자가 한정된 적재 공간에 직육면체 박스형 화물을 어떻게 배치해야 가장 적은 공간으로, 안정적으로, 깨짐주의 제약을 만족하며 실을 수 있는지 계산하고 3D로 확인하는 의사결정 지원 도구다.

권장 문제 정의는 다음과 같다.

> 사용자가 선택한 적재 공간 타입과 박스 목록을 입력하면, 시스템은 회전 가능한 직육면체 박스를 안정성 제약과 깨짐주의 제약을 만족하도록 배치하여 선택 공간의 필요 개수를 최소화한 적재안을 계산하고, 이를 3D 시각화와 추가 적재 시뮬레이션으로 설명한다.

현장 사용 관점의 보조 문제 정의는 다음과 같다.

> 현장 작업자는 태블릿 또는 모바일에서 공간 선택, 박스 입력, 실행 전 확인, 결과 확인, 추가 박스 시뮬레이션, 저장/복원까지 한 기기에서 완료할 수 있어야 한다. 단, 여러 기기 간 자동 이어하기와 팀 공유는 V2 서버 도입 범위로 분리한다.

### 1.2 핵심 용어

- 공간: 파레트, 컨테이너, 트럭 짐칸처럼 짐을 쌓는 제한된 3D 영역.
- 박스: 공간에 적재할 짐. 내부 데이터 모델에서는 기존 `Block` 명칭을 유지할 수 있으나 화면에는 `박스`로 표시한다.
- 박스 유형: 같은 이름, 치수, 깨짐주의 속성을 공유하는 박스 묶음.
- 안전 여유: 실제 적재 가능 치수에서 차감할 여유치. 내부 계산에서는 기존 `offset` 필드를 유지한다.
- 적재 가능 크기: `가로 - 안전 여유 가로`, `세로 - 안전 여유 세로`, `높이 - 안전 여유 높이`로 계산되는 실제 계산 기준 치수.
- 작업본: IndexedDB에 자동저장되는 현재 편집 상태.
- 백업 파일: 사용자가 JSON 파일로 만들어 다른 기기 또는 브라우저에서 가져올 수 있는 이동/복구 파일.
- 동기화: V2에서 서버 저장소를 기준으로 여러 기기 작업 상태를 맞추는 기능.

### 1.3 V1 제품 가정

- 박스는 모두 직육면체다.
- 회전은 90도 단위 직교 회전만 허용한다.
- 한 번의 최적화 실행은 하나의 공간 타입만 사용한다.
- 최적화 목표는 선택한 공간 타입의 필요 개수를 최소화하는 것이다.
- 안정성은 V1에서 기하학적 지지 조건으로 정의한다. 즉, 박스는 바닥 또는 다른 박스 위에 지지되어야 한다.
- 깨짐주의 박스 위에는 일반 박스를 놓지 않는다.
- 깨짐주의 박스끼리의 적층은 허용한다.
- 추가 박스 넣기는 기존 배치를 잠그고 빈 공간만 대상으로 반복 시뮬레이션한다.
- V1은 서버 없는 프론트엔드 단독 구조로 개발한다.
- V1 저장 기본 전략은 `IndexedDB 자동저장 + JSON import/export`다.
- `localStorage`는 테마, 최근 탭, 마지막 선택 같은 비핵심 UI 설정에만 사용한다.
- File System Access API는 Chromium 계열 선택 편의 기능으로만 둔다.
- 태블릿은 터치만으로 전체 플로우를 끝까지 수행 가능해야 한다.
- 모바일은 제약형 end-to-end를 지원한다. 즉, 복잡한 표 편집과 3D 탐색은 단순화 UI로 제공하되 새 작업 생성부터 백업 파일 만들기까지 완료 가능해야 한다.

### 1.4 V1 확정 결정

V1에서는 다음 결정을 확정한다.

- 팔레트 기본값은 요구사항의 `1150 x 1150 x 1550mm`를 유지한다. 표준 팔레트 preset은 후속 확장 항목으로 둔다.
- 컨테이너 preset은 단일 "컨테이너"가 아니라 `20ft GP`를 기본 preset으로 제공한다. `40ft GP`는 후속 확장 항목으로 둔다.
- 트럭 preset 공식 표시명은 `2.5톤반`으로 통일한다. 상세 제원과 출처는 preset 메타데이터에 별도로 보관한다.
- 깨짐주의 박스 위에는 일반 박스를 놓지 않는다.
- 깨짐주의 박스끼리의 적층은 허용한다.
- 추가 박스 체이닝은 `추천 결과 계산`을 기본 액션으로 제공하고, 선택 박스별 `최대`/`수량 지정` 조건을 제공한다.
- 커스텀 공간, 커스텀 박스, 작성 중 draft, 최근 결과, 체이닝 상태는 로컬 작업본과 백업 파일에 보존한다.

### 1.5 V2 현장 피드백 반영 결정

V2에서는 현장 시연 피드백을 기준으로 다음 결정을 반영한다.

- 신규 작업의 기본 파레트는 `기본 파레트`로 표시하며 실제 입고 가능 치수 `1100 x 1100 x 1550mm`, 안전 여유 0을 사용한다.
- 공격 적재 검토용 `오버행 파레트`는 `1150 x 1150 x 1550mm`, 안전 여유 0을 기본 preset으로 제공한다.
- 기존 V1/V1.1 백업 또는 IndexedDB 작업본에 남아 있는 `preset-pallet-1150` 선택값은 `기본 파레트`로 정규화한다.
- 기본 파레트 결과가 오버행 파레트 재계산에서 공간 수 또는 미적재 수를 줄일 수 있으면 결과 화면에 `오버행 파레트 검토` 추천을 표시한다.
- 오버행 추천은 자동 적용하지 않는다. 현장 책임자 확인 후 사용자가 공간 선택에서 오버행 파레트를 직접 선택하고 다시 계산한다.

## 2. 조사 및 접근안 비교

### 2.1 참고 자료 신뢰성 판단

- Hapag-Lloyd Container Specification PDF: 선사 공식 제원 문서이므로 신뢰도가 높다. 다만 제조사/시리얼별 편차가 있고 문서도 sample fleet 성격을 명시하므로 preset에는 `예시치` 배지를 붙이는 것이 안전하다. 20ft GP 내부 치수 후보는 `5,900 x 2,352 x 2,393mm`다.
- Hyundai Mighty 2026-05 카탈로그 PDF: 제조사 공식 자료이므로 트럭 preset 후보의 신뢰도가 높다. 다만 실제 출고 상태와 옵션에 따라 차이가 있을 수 있다고 명시되어 있으므로 출처 배지와 검증일을 함께 표시해야 한다. 트럭 preset의 UI 표시명은 `2.5톤반`으로 고정하고 상세 제원명은 메타데이터로 둔다.
- ISO 3394:2012: ISO 공식 표준이며 2024년에 재확인된 현행 표준이다. 박스/포장 모듈 preset을 설계할 때 `600 x 400`, `600 x 500`, `550 x 366` 계열을 참고하기 적합하다.
- Next.js Static Exports 공식 문서: Next.js는 `output: 'export'`로 정적 HTML/CSS/JS 산출물을 만들 수 있고, 서버가 필요한 기능은 제외된다고 명시한다. V1 프론트 단독 개발의 공식 근거로 적합하다.
- MDN IndexedDB 문서: IndexedDB는 구조화 데이터와 트랜잭션이 필요한 브라우저 저장에 적합하며 Web Worker에서도 사용할 수 있다. V1 작업본 저장소로 적합하다.
- MDN Storage quotas 문서: 브라우저 저장소는 저장 압박, 사생활 보호 모드, 브라우저 정책에 따라 삭제될 수 있으므로 영구 보존을 약속하면 안 된다. V1에서 JSON export/import를 함께 제공해야 하는 근거다.
- MDN File System API 문서: Secure context에서만 일부 브라우저가 지원한다. V1 기본 저장전략이 아니라 선택 편의 기능으로 두는 것이 적절하다.
- W3C WCAG 2.2 Target Size Minimum 문서: 포인터 입력 대상의 최소 크기 기준을 제시한다. 현장 태블릿/모바일에서는 최소 충족보다 넉넉한 44~48px 버튼/입력 높이를 제품 기준으로 둔다.
- Android Views Accessibility 공식 문서와 Apple UI Design Tips: 터치 가능한 컨트롤, 읽기 쉬운 텍스트, 명확한 동작 표현의 근거로 사용한다.

### 2.2 3가지 화면 구성 방법

1. 완전 분리형 멀티페이지
   - 장점: 공간 관리, 박스 관리, 결과 조회가 명확하게 분리된다.
   - 단점: 사용자가 계산 결과까지 도달하기 전에 이동이 많다.
   - 판단: 데이터 관리 기능이 커진 이후 확장안으로 적합하다.

2. 단일 대시보드형 작업대
   - 장점: 반복 계산 속도가 빠르고 고급 사용자가 한 화면에서 조작하기 좋다.
   - 단점: 처음 사용하는 사용자에게 입력 항목과 3D 결과가 동시에 보여 복잡하다.
   - 판단: V1 초기 경험에는 진입 장벽이 높다.

3. 단계형 입력 + 결과 전용 작업대
   - 장점: 입력 정확성을 확보하면서 결과 화면에서는 3D 탐색과 체이닝 반복이 쉽다.
   - 단점: 화면 수가 최소 5개 필요하다.
   - 판단: V1 추천안이다.

### 2.3 모바일/태블릿 UX 접근안

1. 데스크톱 우선 + 모바일은 깨지지 않기만 보장
   - 장점: 구현비가 가장 낮다.
   - 단점: 현장 작업자가 모바일/태블릿으로 실제 입력, 검토, 저장을 완료하기 어렵다.
   - 판단: 최신 사용자 요청을 충족하지 못한다.

2. 모바일 전면 최적화
   - 장점: 현장 UX가 가장 강하다.
   - 단점: 박스 입력 테이블, 3D 조작, 체이닝 UI를 모두 모바일 전용으로 재설계해야 하므로 V1 범위가 커진다.
   - 판단: V2 또는 별도 모바일 제품 전략으로 적합하다.

3. 태블릿 전체 작업 + 모바일 제약형 end-to-end
   - 장점: V1 범위를 유지하면서 현장 사용 가능성을 확보한다.
   - 단점: 모바일에서 3D 탐색과 테이블 편집은 축약 UI가 필요하다.
   - 판단: V1 채택안이다.

### 2.4 개발/저장 접근안

1. 서버 포함 웹 서비스
   - 장점: 멀티디바이스 동기화, 중앙 백업, 팀 공유, 권한 관리가 쉽다.
   - 단점: 인증, 권한, DB 운영, 백업, 보안, 비용이 즉시 발생한다.
   - 판단: V1에는 과하다. V2 트리거가 발생할 때 도입한다.

2. 프론트 단독 + localStorage
   - 장점: 구현이 가장 단순하다.
   - 단점: 용량, 트랜잭션, 구조화 데이터, 안정성 측면에서 커스텀 공간/박스 저장소로 부적합하다.
   - 판단: UI 설정 저장 정도로 제한한다.

3. 프론트 단독 + IndexedDB 작업본 + 백업 파일
   - 장점: 구조화 데이터 자동저장과 사용자 주도 백업/기기간 이동을 동시에 제공한다.
   - 단점: 자동 멀티디바이스 동기화는 제공하지 못한다.
   - 판단: V1 채택안이다.

## 3. 제품 범위 및 로드맵

### 3.1 V1 범위

- Next.js 기반 프론트엔드 단독 애플리케이션.
- 정적 export 가능한 구조를 기본으로 한다.
- IndexedDB에 커스텀 공간, 커스텀 박스, draft, 최근 결과, 체이닝 상태를 자동저장한다.
- 백업 파일 가져오기/만들기로 다른 기기 또는 브라우저에 작업 데이터를 이동할 수 있게 한다.
- 저장된 공간/박스 라이브러리를 유지한 채 현재 작업 박스, 결과, 추가 적재 기록만 비우는 `새 작업 시작`을 제공한다.
- 저장된 박스는 이름, 치수, 깨짐주의 여부로 검색해 현재 작업에 다시 추가할 수 있게 한다.
- 현장 시연을 빠르게 시작할 수 있도록 현재 작업에 `2.5톤반` 기준 예제 박스를 불러오는 경로를 제공한다.
- 결과가 아슬아슬하게 여러 공간으로 나뉠 때 안전 여유 조정 검토값을 추천하되, 자동 적용하지 않고 현장 책임자 확인 후 수정하도록 안내한다. 추천값은 별도 미리보기에서 실제 수정 없이 3D로 확인할 수 있다.
- V2 기본 파레트 결과에서 오버행 파레트로 공간 수 또는 미적재 수가 줄어드는 경우 `오버행 파레트 검토` 추천을 표시한다. 추천은 실제 선택을 바꾸지 않으며, 별도 미리보기에서 재계산 결과를 확인할 수 있다.
- 결과 화면에는 `현장 전달 전 점검`을 제공해 최신 결과, 미적재/경고, 백업 파일 상태를 한 번에 확인하게 한다.
- 태블릿에서 전체 플로우를 터치만으로 수행할 수 있게 한다.
- 모바일에서 공간 선택, 박스 입력, 실행 전 확인, 결과 요약 확인, 백업 파일 만들기까지 완료 가능하게 한다.
- V1 결과 구현은 계산 엔진 좌표를 3D 적재 뷰로 우선 검토하고, 위/앞/옆 2D 보기를 보조 검토 모드로 유지한다.
- 결과 3D는 데스크톱/태블릿에서 탐색형, 모바일에서 버튼 중심 검토형으로 제공한다.
- 저장 상태를 `저장 중`, `자동저장됨`, `저장 실패`, `백업 필요`, `다른 탭 저장 감지`로 표시한다.

### 3.2 V1.1 확장 범위

- 로컬 작업본 보호 UX 1차: `navigator.storage.persist()` 요청, 저장 용량 안내, 마지막 내보내기 시각, 내보내기 리마인더를 먼저 제공한다.
- PWA 오프라인 재진입 1차: 서비스워커 앱 셸 캐시와 저장 보호 패널의 `오프라인 준비` 상태를 제공한다.
- 동일 기기 다중 탭 충돌 방지.
- 백업 파일 가져오기 충돌 처리 모달.
- 로컬 스키마 마이그레이션 실패 시 복구 안내.
- 결과 3D 렌더링 1차: Three.js 기반 3D 적재 뷰, 2D 보조 뷰 전환, 박스 유형 강조 동기화.

### 3.2.1 로컬 작업본 보호 UX 1차 결정

이번 증분은 서버 없이 프론트 단독으로 처리한다.

- 자동저장은 IndexedDB 작업본을 대상으로 하며, 브라우저 데이터 삭제나 기기 분실까지 보장하지 않는다.
- 백업 파일은 다른 기기 이동과 복구를 위한 실제 백업 수단으로 안내한다.
- 상단 저장 상태 pill은 저장 보호 패널을 여는 버튼으로 동작한다.
- 저장 보호 패널은 `이 기기 저장`, `백업 파일`, `브라우저 보호` 3개 상태만 보여준다.
- 의미 있는 사용자 데이터가 생기기 전에는 내보내기 리마인더를 띄우지 않는다.
- 커스텀 공간, 박스 템플릿, 현재 작업 박스, 결과, 체이닝 이력 중 하나라도 생기고 백업 파일이 최신이 아니면 `백업 업데이트 필요`를 표시한다.
- `navigator.storage.persist()`는 사용자가 직접 `작업 보호 강화`를 누를 때만 요청한다.
- Storage API 미지원, 비보안 컨텍스트, 브라우저 거절은 시스템 오류가 아니라 `백업 파일 권장` 안내 상태로 처리한다.
- PWA manifest와 icon은 설치 준비 메타데이터로 유지한다.
- 서비스워커 앱 셸 캐시는 한 번 연 화면의 재진입을 돕는 보조 안전장치로만 안내하고, 백업 파일을 대체한다고 표현하지 않는다.

### 3.2.2 PWA 오프라인 재진입 1차 결정

이번 증분은 서버 없이 프론트 단독으로 처리한다.

- 서비스워커는 루트 화면, manifest, icon, Next.js 앱 셸 asset을 캐시한다.
- navigation 요청은 network-first로 처리하고, 네트워크 실패 시 캐시된 루트 화면을 반환한다.
- 저장 보호 패널에 `오프라인 준비` 행을 추가해 `확인 중`, `준비 중`, `준비됨`, `준비 실패`, `지원되지 않음` 상태를 보여준다.
- 화면 문구는 `서비스워커`, `캐시` 같은 기술어보다 `오프라인 준비`, `인터넷이 끊겨도 앱 화면을 다시 여는 데 도움`처럼 현장 언어를 우선한다.
- 오프라인 준비는 작업 데이터 보존을 보장하지 않으므로, 중요한 작업은 계속 `백업 파일 만들기`로 보관해야 한다.
- PWA 설치 안내는 저장 보호 패널의 `앱 설치` 행으로 제공한다.
- `beforeinstallprompt` 이벤트가 있으면 사용자가 `앱 설치` 버튼을 눌렀을 때 브라우저 설치 확인 창을 연다.
- 설치 이벤트가 없는 브라우저는 Chrome/Edge의 주소창 설치 아이콘 또는 메뉴, Safari의 공유 버튼 `홈 화면 추가` 경로를 안내한다.
- 설치 여부와 작업 데이터 보존은 별개이므로, 설치된 상태에서도 백업 파일 만들기 안내를 유지한다.

### 3.2.3 동일 기기 다중 탭 충돌 방지 결정

이번 증분은 같은 브라우저 origin 안에서 동일 작업본을 여러 탭으로 열었을 때 stale 탭이 최신 IndexedDB 작업본을 덮어쓰지 못하게 하는 범위다. 여러 기기 간 동기화는 여전히 V2 서버 범위다.

- IndexedDB 저장소는 `expectedRevision` 기반 optimistic guard를 사용한다.
- 같은 `fileId`의 저장소 revision이 현재 탭의 마지막 저장 기준 revision보다 크면 자동저장을 차단한다.
- BroadcastChannel은 다른 탭 열림, 다른 탭 저장 완료, 탭 종료 신호를 전달하는 실시간 UX 신호로 사용한다.
- BroadcastChannel 미지원 환경에서는 localStorage에 같은 메시지 payload를 기록하고 `storage` event를 보조 신호로 사용한다.
- `storage` event는 UX fallback일 뿐이며, 데이터 정합성은 IndexedDB revision guard가 보장한다.
- Web Locks API는 단일 편집권을 강제하는 후속 enhancement 후보로 남긴다. 이번 증분에서는 stale state 자체를 해결하지 못하므로 correctness 기반에서 제외한다.
- 다른 탭 저장이 감지된 탭은 `읽기 전용` 상태로 전환하고 입력 변경, 결과 생성, 체이닝 확정을 막는다.
- 사용자는 `최신 작업본 불러오기`로 IndexedDB 최신본을 다시 로드하거나, `현재 작업 백업 만들기`로 충돌 탭의 현재 화면 상태를 파일로 남길 수 있다.
- 저장 보호 패널은 행을 늘리지 않고 `이 기기 저장` 행 안에서 `이 탭이 편집 중`, `이 탭이 편집 중 · 다른 탭 열림`, `다른 탭 최신본 감지`를 표시한다.

### 3.2.3 결과 3D 렌더링 1차 결정

이번 증분은 결과 화면의 중앙 뷰어를 `3D 적재 뷰` 기본값으로 전환하고, 기존 2D 투영은 같은 슬롯 안의 보조 모드로 유지한다.

- 3D 렌더링은 `plain three` 기반 client-only 컴포넌트로 구현한다.
- `ResultStage`는 공간 선택, 보기 모드, 카메라 프리셋, 박스 유형 선택 상태만 관리한다.
- `PackedBlock`의 회전 후 최종 치수와 `x/y/z` 좌표를 Three.js y-up 중심 좌표로 변환하는 순수 유틸을 둔다.
- 2D 투영과 3D는 같은 박스 유형 색상 팔레트를 공유한다.
- 기본 카메라는 자유시점으로 시작하며, 사용자는 `3D`, `위`, `앞`, `옆`, `처음` 버튼으로 검토 모드를 전환한다.
- 3D 모드에서는 `자유시점`, `위`, `앞`, `옆` 카메라 프리셋을 제공한다.
- 범례 선택과 3D 박스 클릭은 모두 `blockTemplateId` 기준의 박스 유형 강조로 동기화한다.
- 추가 박스 시뮬레이션 preview가 있을 때 3D에서도 기존 박스와 신규 박스의 시각 상태를 구분한다.
- WebGL 초기화 실패 시 2D 보조 뷰로 돌아갈 수 있도록 2D 투영을 계속 유지한다.
- 모바일 `크게 보기`와 치수 오버레이는 V1 결과 확인 흐름에 포함한다.
- 절단면, 폭발도, 수동 배치 편집은 후속 범위로 남긴다.

### 3.3 V2 확장 범위

- 사용자 계정 또는 조직/작업공간 기반 서버 저장소.
- 여러 기기 간 자동 동기화.
- 기기 교체 후 자동 복구.
- 팀 공유 preset과 권한 관리.
- 감사 이력, 변경 이력, 승인 흐름.
- 오프라인 편집 후 재연결 동기화와 충돌 해결 큐.

### 3.4 서버 도입 트리거

다음 조건 중 하나라도 제품 요구로 확정되면 서버 도입을 검토가 아니라 필수 범위로 전환한다.

- JSON 파일 없이 다른 기기에서 이어서 작업해야 한다.
- 브라우저 데이터 삭제, 기기 분실 이후에도 커스텀 공간/박스를 자동 복구해야 한다.
- 공용 preset, 공용 박스 라이브러리, 팀 협업, 승인 기능이 필요하다.
- 동일 사용자의 2개 이상 기기 사용이 일반 사용 패턴이 된다.
- 커스텀 공간/박스가 운영상 손실 불가 자산으로 간주된다.
- 고객 지원팀이 사용자 파일 없이 데이터를 복구해야 한다.

## 4. 추천 정보 구조

### 4.1 화면 IA

1. 공간 선택
2. 박스 등록
3. 실행 전 확인
4. 결과 3D
5. 추가 박스 시뮬레이션

초기 V1은 위 5개 화면을 핵심 흐름으로 만든다. 각 단계는 선형 진행을 기본으로 하며, 화면 이동은 하단 CTA로 수행한다. 별도 메뉴로서의 공간 라이브러리, 박스 라이브러리, 결과 조회는 이후 확장한다.

모든 화면에는 공통 저장 상태 바를 둔다.

- 데스크톱/태블릿: 상단 우측에 저장 상태와 import/export 버튼을 표시한다.
- 모바일: 하단 sticky action 영역 위에 저장 상태를 축약 표시한다.
- 저장 상태 pill을 선택하면 자동저장, 백업 파일 최신성, 브라우저 보호 상태를 한 패널에서 확인한다.
- 저장 실패 시 현재 화면을 떠나지 않고 `백업 파일 만들기`, `복사본 만들기`, `불필요한 작업 삭제` CTA를 제공한다.
- 다른 탭 저장 충돌 시 저장 상태 pill은 red tone의 `다른 탭 저장 감지 · 읽기 전용`을 표시하고 저장 보호 패널을 자동으로 연다.
- 충돌 상태의 primary CTA는 `최신 작업본 불러오기`이며, secondary CTA는 `현재 작업 먼저 백업`이다.
- 다른 탭이 열려 있지만 충돌은 아닌 경우 `이 기기에 저장됨 · 다른 탭 열림`을 비차단 안내로 표시한다.

### 4.2 핵심 유저 플로우

1. 사용자는 새 적재 계산을 시작한다.
2. 기본 공간 또는 내 공간을 선택한다.
3. 시스템은 외형 치수, 안전 여유, 적재 가능 크기, 출처를 보여준다.
4. 사용자는 박스 유형을 추가하고 이름, 치수, 수량, 깨짐주의 여부를 입력한다.
5. 시연 담당자는 필요할 때 `시연 예제 불러오기`로 2.5톤반 기준 예제 박스를 현재 작업에 채울 수 있다.
6. 시스템은 총 박스 수, 총 부피, 공간 1개 적재 가능 부피, 예상 최소 공간 수 하한을 계산한다.
7. 사용자는 실행 전 확인에서 제약과 누락값을 확인한다.
8. 사용자는 시뮬레이션 실행을 누른다.
9. 결과 화면에서 공간별 2D 보조 배치, 적재율, 남은 부피, 미적재 박스, 실행 시간을 확인한다.
10. 박스 유형을 클릭하면 결과 작업대에서 해당 박스 유형만 강조된다.
11. 사용자는 추가 박스 시뮬레이션으로 진입해 기존 배치를 잠근 상태에서 빈 공간 추가 적재를 시뮬레이션한다.
12. 추가 결과를 확정하면 체이닝 이력이 누적된다.
13. 시스템은 작업본을 IndexedDB에 자동저장하고 마지막 저장 시각을 갱신한다.
14. 사용자는 JSON으로 작업을 내보내 다른 기기에서 가져올 수 있다.

### 4.3 Stepper 규칙

- Stepper는 내비게이션이 아니라 현재 단계와 전체 단계 수를 알려주는 진행 표시다.
- 각 화면 제목 바로 위에 `N / 5 + 단계명`을 함께 표기한다.
- 완료 단계, 현재 단계, 대기 단계를 시각적으로 구분한다.
- 대기 단계는 아직 도달하지 않은 단계이지, 비활성 오류 상태처럼 보이면 안 된다.
- Stepper 다음 포커스는 화면 제목으로 이동한다.

## 5. 화면별 기획

### 5.1 공간 선택

목적: 사용자가 계산 기준이 되는 적재 공간을 정확히 선택한다.

레이아웃:

- 좌측: 기본 공간 리스트, 최근 사용, 내 공간 탭.
- 중앙: 선택 공간 상세. 이름, 유형, 외형 치수, 안전 여유, 적재 가능 크기.
- 우측: 치수 다이어그램 또는 미니 3D preview.
- 하단: `다음: 박스 등록` CTA.

주요 컴포넌트:

- 공간 카드
- 출처 배지
- 치수 입력 폼
- 안전 여유 입력 폼
- 적재 가능 크기 계산 결과
- custom 공간 저장 버튼
- 저장 상태 바

필수 상태:

- 선택 전 빈 상태.
- 기본 공간 로딩.
- custom 저장 중.
- custom 저장 실패.
- IndexedDB 자동저장 실패.
- 적재 가능 크기 0 이하 오류.

CTA 규칙:

- `다음`은 공간이 선택되고 적재 가능 크기가 0보다 클 때만 활성화한다.
- disabled 상태에서는 버튼 아래에 `공간을 선택하고 적재 가능 크기를 확인해야 합니다`를 표시한다.

접근성 주석:

- 오류는 필드 옆 텍스트와 상단 요약 경고에 동시에 반영한다.
- 출처 배지는 색상만 쓰지 않고 `공식`, `예시치`, `검증일` 텍스트를 함께 표시한다.

### 5.2 박스 등록

목적: 사용자가 적재할 박스 유형과 수량, 깨짐주의 여부를 입력한다.

레이아웃:

- 본문: 박스 유형 카드 목록.
- 우측: 선택 행 상세 편집과 입력 요약.
- 하단: `다음: 실행 전 확인` CTA.

모바일 레이아웃:

- 테이블을 카드형 리스트로 전환한다.
- 각 박스 유형은 접이식 카드로 보여준다.
- 숫자 입력은 `inputmode=numeric`을 사용한다.
- 깨짐주의 토글은 텍스트 라벨을 함께 표시한다.

테이블 컬럼:

- 박스명
- 가로(mm)
- 세로(mm)
- 높이(mm)
- 수량
- 깨짐주의 여부
- 단위 부피
- 총 부피

필수 상태:

- 박스 0개 빈 상태.
- 행 단위 validation 오류.
- 중복 이름 경고.
- 수량 0 경고.
- preset 불러오기 실패.
- 자동저장 실패.

CTA 규칙:

- `다음`은 박스 유형이 1개 이상이고 모든 필수값이 유효할 때만 활성화한다.
- 오류가 있는 행은 인라인 에러와 상단 요약 경고에 동시에 반영한다.

접근성 주석:

- 테이블 셀 편집은 Tab/Shift+Tab으로 이동한다.
- Enter는 같은 행 저장, Esc는 편집 취소다.
- 오류 행 수정 후에는 해당 행 첫 셀로 포커스를 돌려 사용자가 맥락을 잃지 않게 한다.

### 5.3 실행 전 확인

목적: 계산 전에 입력값과 제약을 사용자가 확인하고 오류를 줄인다.

레이아웃:

- 상단 요약: 선택 공간, 총 박스 수, 총 부피, 공간 1개 적재 가능 부피, 예상 최소 공간 수 하한.
- 본문: 제약 체크리스트.
- 우측: 경고/위험 패널.
- 하단: `시뮬레이션 실행` CTA.

체크리스트:

- 안전 여유 적용 완료.
- 모든 박스 치수/수량 유효.
- 깨짐주의 규칙 적용.
- 깨짐주의끼리 적층 허용.
- 깨짐주의 위 일반 박스 적층 금지.
- 직육면체/직교 회전만 지원.
- 1회 실행당 하나의 공간 타입 사용.
- 작업본 자동저장 완료 또는 백업 파일 만들기 가능.

필수 상태:

- 실행 가능.
- 경고 포함 실행 가능.
- 실행 불가.
- 계산 준비 중.
- 실행 실패 후 재시도.

CTA 규칙:

- `시뮬레이션 실행`은 오류가 남아 있으면 비활성화한다.
- 경고만 있으면 활성화하되, 경고 영역에 계산 리스크를 명시한다.
- 실행 중에는 버튼 라벨을 `계산 중...`으로 바꾸고 spinner를 표시한다.

접근성 주석:

- 상태 변경 문구는 `role=status`로 읽히게 한다.
- 실패 문구는 원인과 재시도 방법을 텍스트로 설명한다.

### 5.4 결과 3D

목적: 사용자가 최적화 결과를 공간별로 이해하고 검증한다.

V1 구현 기준:

- 계산 엔진이 생성한 `x/y/z` 좌표와 회전 후 치수를 기준으로 3D 적재 뷰를 제공한다.
- 위, 앞, 옆 2D 보기는 WebGL fallback과 정밀 검토를 위한 보조 모드로 유지한다.
- 결과 생성 시점의 공간 스냅샷을 결과에 저장해, 사용자가 이후 공간 선택을 바꿔도 과거 결과의 투영 기준이 흔들리지 않게 한다.
- 좌측은 공간 인스턴스 목록, 중앙은 3D/2D 배치 뷰어, 우측은 박스 유형 범례로 구성한다.
- 범례, 2D 보드 박스, 3D 박스를 선택하면 해당 박스 유형을 강조하고 나머지는 낮은 불투명도로 유지한다.
- 3D 뷰를 기본으로 제공하되, 현장 검토 UX는 2D 보조 모드에서도 end-to-end로 동작해야 한다.

레이아웃:

- 상단 요약 바: 사용 공간 수, 평균 적재율, 남은 부피, 미적재 박스 수, 실행 시간, 제약 위반 여부.
- 좌측: 공간 인스턴스 탐색. Space 1, Space 2, 미적재. 추가 시뮬레이션 중에는 결과 뷰어 안에 추가 결과 미리보기 안내와 비교 버튼을 표시한다.
- 중앙: 3D 캔버스.
- 우측: 박스 유형 범례, 선택 상태, 입력 수정, 다시 계산, 추가 박스 시뮬레이션.

모바일 레이아웃:

- 상단에 결과 KPI를 2열 카드로 제공한다.
- 모바일 3D는 자유 조작보다 `3D`, `위`, `앞`, `옆`, `처음` 버튼 중심의 검토 모드로 시작한다.
- 범례와 선택 상태는 하단 영역으로 전환한다.
- 백업 파일 만들기 CTA를 결과 요약 근처에 노출한다.

3D 표현 규칙:

- 기본 카메라는 등각 시점으로 시작한다.
- 뷰 툴바는 3D, 위, 앞, 옆, 처음을 제공한다.
- 3D 카메라 툴바는 자유시점, 위, 앞, 옆 프리셋을 제공한다.
- `방향 표시` 토글은 박스가 처음 입력한 높이 방향을 화살표로 보여주며, 복잡한 결과에서는 사용자가 숨길 수 있다.
- active 예시는 툴바에 명확히 표시한다.
- 범례에서 박스 유형을 선택하면 해당 유형은 100% 불투명, 나머지는 20~30% 투명으로 표시하되 윤곽선은 유지한다.
- 캔버스 박스 클릭과 범례 선택은 양방향 동기화한다.
- 3D 캔버스 안에는 선택 공간의 가로, 깊이, 높이를 mm 단위로 표시하는 치수 오버레이를 둔다.
- tooltip은 박스명, 치수, 깨짐주의 여부, 입력 높이 방향을 우선 보여주고 좌표 상세는 후속으로 확장한다.

필수 상태:

- 결과 생성 중.
- 정상 결과.
- 부분 성공.
- 미적재 박스 있음.
- 계산 실패.
- 저장 실패.
- 내보내기 필요.

CTA 규칙:

- `입력 수정`은 항상 활성화한다.
- `다시 계산`은 결과 또는 실패 후 활성화한다.
- `추가 박스 시뮬레이션`은 결과가 있을 때만 활성화한다.
- `백업 파일 만들기`는 결과 여부와 관계없이 작업본이 있을 때 활성화한다.

접근성 주석:

- 3D 캔버스는 키보드 포커스가 가능해야 한다.
- tooltip은 hover와 focus 모두에서 제공한다.
- 색상만으로 깨짐주의/선택 상태를 표현하지 않고 라벨, 아이콘, 패턴을 함께 사용한다.

### 5.5 추가 박스 시뮬레이션

목적: 최종 결과를 기준으로 남은 빈 공간에 특정 박스 유형을 얼마나 더 넣을 수 있는지 반복 확인한다.

V1 구현 기준:

- 추가 박스 시뮬레이션은 새 공간을 만들지 않고 현재 결과 공간 내부의 남은 빈 공간만 대상으로 계산한다.
- 선택 가능한 박스는 저장된 박스 라이브러리 전체를 대상으로 하되, 검색/상위그룹/하위그룹으로 좁혀 찾는다.
- 검색 결과가 6개를 넘으면 본문에 펼치지 않고 `저장된 박스 찾아 선택` 모달에서 페이지 단위로 선택한다.
- `추천 결과 계산`을 기본 액션으로 제공하고, 선택한 박스별로 `최대` 또는 `수량 지정` 조건을 설정할 수 있게 한다.
- preview 상태에서는 기존 박스를 낮은 불투명도로, 신규 추가 박스를 100% 불투명도와 dashed outline으로 구분한다.
- 확정 시 결과 좌표와 평균 적재율을 갱신하고 chain history에 직전 결과 스냅샷을 저장해 1단계 undo를 지원한다.

레이아웃:

- 결과 2D 작업대 바로 아래: 추가 박스 시뮬레이션 full-width panel.
- 상단: 기준 결과 잠금 배지, 간단한 chain chip row.
- 좌측: 저장된 박스 검색, 그룹 필터, 6개 이하 직접 선택 또는 대량 선택 모달.
- 우측: 박스별 추가 조건, 추천 결과 비교, 이번 단계 가능 수량, 확정, 직전 추가 취소.

상태 머신:

1. 선택 전: 추가할 박스 유형을 선택하기 전 상태.
2. 계산 중: 추천 결과 계산이 실행된 상태.
3. preview: 신규 박스 배치가 100% 불투명으로 미리 보이는 상태.
4. 추가 가능 0: 실패가 아니라 정상 계산 결과로 분리한다.
5. 실패: 계산 실패 또는 제약 위반.
6. 확정: preview 결과를 체인 이력에 추가한 상태.
7. 되돌림: 직전 확정 단계를 제거한 상태.

CTA 규칙:

- `추천 결과 계산`은 박스 선택 후 활성화한다.
- 박스별 `수량 지정`은 1개 이상의 정수만 허용하며, 비활성 상태의 박스는 `최대` 조건으로 계산한다.
- `확정`은 preview 성공일 때만 활성화한다.
- `되돌리기`는 체이닝 이력이 있을 때만 활성화한다.

접근성 주석:

- 상태 전환마다 live region 알림을 제공한다.
- `추가 가능 0`은 실패 색이 아니라 중립 상태로 표시한다.
- `되돌리기`는 destructive action이 아니라 history rollback이므로 warning 색이 아니라 neutral secondary 버튼으로 둔다.

### 5.6 백업 파일 가져오기/만들기

목적: 서버 없는 V1에서도 사용자가 커스텀 데이터와 작업 상태를 보존하고 다른 기기에서 복원할 수 있게 한다.

백업 파일 포함 범위:

- `schema_version`
- `app_version`
- `exported_at`
- `device_id`
- `file_id`
- `revision`
- `policy.fragile_stack_on_fragile_allowed`
- `policy.truck_preset_display_name = "2.5톤반"`
- custom spaces
- custom blocks
- draft
- recent results
- chain history

가져오기 충돌 처리:

- 다른 `file_id`: `새 작업으로 열기`를 기본으로 한다.
- 같은 `file_id`이고 현재 작업이 수정됨: `현재 작업 유지`, `가져온 파일로 대체`, `복사본으로 열기`, `취소`를 제공한다.
- 스키마 버전이 낮음: 마이그레이션 후 가져오며 원본 파일 백업 안내를 표시한다.
- 스키마 버전이 높음: 가져오기를 막고 최신 앱 필요 안내를 표시한다.

실패 상태:

- JSON 파싱 실패.
- 스키마 불일치.
- 파일 크기 초과.
- unknown key 포함.
- 숫자 범위 오류.
- 저장소 용량 부족.

보안/무결성 규칙:

- import 데이터는 전부 비신뢰 입력으로 취급한다.
- 박스명/공간명은 plain text로 렌더링한다.
- `__proto__`, `constructor`, `prototype` 키는 거부한다.
- 침묵 덮어쓰기와 침묵 삭제는 금지한다.

| CTA | Enabled | Disabled | Loading | Error 후 복구 |

공통 표기 규칙:

- disabled 상태는 클릭 불가 이유를 버튼 아래 짧은 문장으로 함께 표기한다.
- loading 상태는 버튼 라벨 자체를 바꾼다.
- error 후에는 같은 위치에 `재시도` 또는 `입력 수정` 보조 CTA를 노출한다.

## 6. 접근성 및 사용성 기준

- 포커스 순서는 `Stepper -> 화면 제목 -> 주요 입력 영역 -> 우측 요약 -> 하단 CTA`를 따른다.
- 모든 버튼은 visible focus를 보장한다.
- 오류 메시지는 텍스트로 원인을 설명한다.
- 상태 메시지는 live region으로 알린다.
- 테이블 입력은 Tab/Shift+Tab으로 이동한다.
- Enter는 같은 행 저장, Esc는 편집 취소로 동작한다.
- 3D 캔버스는 키보드 포커스가 가능해야 한다.
- 3D 키보드 조작은 화살표 회전, +/- 줌, 1/2/3 시점 전환, 0 처음 보기, Esc 선택 해제를 제공한다.
- 3D 터치 조작에는 버튼 기반 대체 컨트롤을 제공한다.
- 깨짐주의/선택 상태는 색상만으로 표현하지 않고 라벨, 아이콘, 패턴을 함께 사용한다.
- 터치 타깃은 WCAG 2.2 최소 기준을 하한으로 두되, 제품 주요 버튼과 입력은 44px 이상을 기본으로 한다.
- 모바일 현장 주요 CTA는 장갑 착용과 흔들림을 고려해 48px 이상 높이와 충분한 간격을 둔다.
- 계산 시간이 긴 경우 `검증 중 -> 적재 계산 중 -> 3D 생성 중`으로 진행 상태를 나누어 표시한다.
- 저장 상태는 항상 사용자가 이해할 수 있는 문장으로 표시한다.

### 6.1 Responsive Rule

- 1280px 이상: 3열 레이아웃을 사용한다.
- 768~1279px: 2열 레이아웃을 사용하고 우측 상세 패널은 본문 아래 또는 접이식 패널로 이동한다.
- 360~767px: 1열 스택으로 전환한다.
- 320~359px: 핵심 정보만 유지하는 압축 레이아웃을 제공한다.
- 모바일에서는 좌/우 패널을 본문 아래로 접고 CTA는 하단 sticky footer로 고정한다.
- sticky footer는 입력 필드, 오류 메시지, 저장 상태를 가리면 안 된다.
- 모바일에서 표는 카드형 입력 리스트로 전환한다.
- 모바일 결과 화면은 조작형 3D보다 KPI, 뷰 버튼, 범례, 백업 파일 만들기 접근성을 우선한다.
- 태블릿은 현장 주 사용 기기로 간주하고 3D 검토, 박스 강조, 추가 박스 preview를 터치만으로 수행 가능해야 한다.

## 7. 인수 기준

1. 사용자가 기본 공간을 선택하면 외형 치수, 안전 여유, 적재 가능 크기가 모두 표시된다.
2. 트럭 기본 공간 공식 표시명은 모든 문서와 UI에서 `2.5톤반`으로 표시된다.
3. 내 공간은 가로/세로/높이/안전 여유가 모두 유효할 때만 저장된다.
4. 박스 유형이 1개 이상이고 모든 수량이 1 이상이어야 실행 전 확인을 통과할 수 있다.
5. 실행 전 확인 화면은 깨짐주의끼리 적층 허용, 깨짐주의 위 일반 박스 적층 금지, 안전 여유 적용, 직교 회전 제약을 명시한다.
6. 결과 화면은 공간별 적재율, 남은 부피, 사용 공간 수, 실행 시간, 미적재 수를 표시한다.
7. 결과 화면은 3D/2D 보기, 입력 수정, 다시 계산, 추가 결과 미리보기 비교 버튼을 포함한다.
8. 박스 유형 클릭 시 해당 유형만 강조되고 다른 유형은 투명 처리되며 윤곽선은 유지된다.
9. 추가 박스 시뮬레이션은 기존 배치를 잠근 상태에서 빈 공간만 대상으로 계산한다.
10. 추가 적재 가능 수량이 0이면 실패가 아닌 정상 결과로 0을 명시하고 기존 결과를 유지한다.
11. 추가 결과 확정 후 다시 추가 박스 시뮬레이션을 반복할 수 있다.
12. 계산 실패 시 브라우저 계산 중단, 메모리 부족, 입력 제약 위반 등 사용자에게 유효한 실패 원인을 보여준다.
13. 커스텀 공간/박스 생성·수정·삭제 후 IndexedDB 자동저장이 동작하고 마지막 저장 시각이 표시된다.
14. 새로고침 또는 같은 기기 재방문 후 커스텀 공간, 커스텀 박스, draft가 복구된다.
15. 백업 파일만으로 동일 데이터 세트를 같은 기기 또는 다른 기기에 복원할 수 있다.
16. 백업 파일 가져오기 충돌 시 현재 작업 유지, 가져온 파일로 대체, 복사본으로 열기, 취소 중 하나를 선택할 수 있다.
17. 저장 실패 시 백업 파일 만들기 또는 복사본 만들기 경로를 제공한다.
18. 사용자는 현재 저장 방식이 로컬 기기 저장인지 즉시 알 수 있다.
19. 태블릿에서 터치만으로 공간 선택, 박스 등록, 실행 전 확인, 결과 확인, 추가 박스 시뮬레이션, 저장/복원이 완료된다.
20. 모바일 360px에서 공간 선택, 박스 등록, 실행 전 확인, 결과 요약 확인, 백업 파일 만들기가 완료된다.
21. 어떤 핵심 기능도 hover, 키보드, 멀티터치 제스처만으로만 동작하지 않는다.
22. 320px, 360px, 390px, 768px, 1024px, 1440px 기준에서 핵심 정보와 CTA가 겹치거나 사라지지 않는다.
23. 동일 기기에서 두 탭을 열면 다른 탭 열림 상태가 저장 상태 영역 또는 저장 보호 패널에 표시된다.
24. stale 탭의 자동저장은 IndexedDB revision guard에서 거부되고 최신 작업본을 조용히 덮어쓰지 않는다.
25. 다른 탭 저장 충돌을 감지한 탭은 읽기 전용 상태로 전환되고 `최신 작업본 불러오기`, `현재 작업 먼저 백업` 경로를 제공한다.
26. 결과가 있으면 중앙 배치 뷰어의 기본 모드는 3D 적재 뷰다.
27. 사용자는 3D, 위, 앞, 옆, 처음 버튼으로 결과 검토 모드를 바꿀 수 있다.
28. 3D 박스 클릭과 범례 선택은 같은 박스 유형 강조 상태를 공유한다.
29. 3D 장면은 공간 인스턴스 전환과 추가 박스 preview 상태를 반영한다.
30. WebGL 렌더링이 실패해도 2D 보조 뷰로 결과 검토를 계속할 수 있다.
31. 안전 여유 조정 추천은 미적재 없이 공간 수가 줄어드는 경우에만 표시하고, 자동 적용하지 않는다.
32. 추천 미리보기는 추천 offset 재계산 결과를 dialog 안에서 보여주되, 저장된 공간과 현재 입력을 변경하지 않는다.
33. 오버행 파레트 검토 추천은 기본 파레트 결과가 오버행 파레트에서 공간 수 또는 미적재 수를 줄일 때만 표시하고, 자동 적용하지 않는다.
34. 오버행 추천 미리보기는 오버행 파레트 재계산 결과를 dialog 안에서 보여주되, 저장된 공간과 현재 선택을 변경하지 않는다.

## 9. Next.js 개발 첫 구현 범위

기획서 반영 후 개발은 전체 최적화 엔진을 한 번에 완성하는 방식이 아니라, V1의 데이터 보존과 현장 입력 골격을 먼저 구현한다.

첫 구현 범위:

- Next.js App Router 기반 정적 프론트엔드 골격.
- `output: 'export'` 가능한 설정.
- 커스텀 공간 CRUD.
- 커스텀 박스 CRUD.
- 저장된 박스 검색과 재사용.
- draft 상태 저장.
- 저장 라이브러리를 보존하는 현재 작업 새로 시작.
- IndexedDB 자동저장.
- 백업 파일 가져오기/만들기.
- 저장 상태 표시.
- 모바일/태블릿 반응형 입력 UI.
- 실행 전 확인과 결과 대기 상태.
- Web Worker 기반 적재 계산.
- 적재 엔진 안전 게이트: 공중 부양, 공간 경계 초과, 박스 충돌, 깨짐주의 정책, 지지면 검증 자동 테스트.
- 추가 박스 체이닝 계산과 추가 결과 미리보기 비교 버튼.
- 결과 및 추가 결과 미리보기 기준의 안전 여유 조정 추천 카드와 추천 적용 3D 미리보기.
- 현장형 preset 적재 정합성/성능 audit.
- 현장 PC 재측정용 `npm run field:audit` 명령.
- 저장 신뢰성 UX: 저장 보호 패널, 백업 파일 리마인더, 동일 기기 다중 탭 충돌 방지.
- 결과 3D 렌더링 1차: Three.js 3D 적재 뷰, 2D 보조 뷰 전환, 박스 유형 강조 동기화.
- PWA 오프라인 재진입 1차: 서비스워커 앱 셸 캐시, 저장 보호 패널 `오프라인 준비` 상태.
- PWA 설치 안내: 저장 보호 패널 `앱 설치` 상태, 브라우저 설치 prompt, Chrome/Edge/Safari 홈 화면 추가 안내.

후속 구현 범위:

- 절단면, 폭발도, 수동 배치 편집.
- 서비스워커 캐시 업데이트 정책 고도화와 완전한 오프라인 시연 시나리오.
- 서버 기반 여러 기기 자동 동기화와 충돌 해결 큐.

## 10. 개선 제안 및 반영

검토 후 개선해야 할 사항은 다음 3가지다.

1. 저장 신뢰성 개선: IndexedDB 자동저장만으로 영구 보존을 약속하지 않고 백업 파일을 함께 제공한다.
2. 현장 UX 개선: 태블릿 전체 작업과 모바일 제약형 end-to-end를 V1 기준으로 반영한다.
3. 확장성 개선: 자동 멀티디바이스 동기화는 V2 서버 도입 트리거로 명확히 분리한다.

이번 최종 기획서에는 세 가지를 모두 반영한다. 특히 PM 관점에서 `프론트 단독 V1`, `IndexedDB 작업본`, `백업 파일`, `2.5톤반 명칭`, `깨짐주의끼리 적층 허용`, `모바일/태블릿 지원 기준`, `V2 서버 도입 조건`을 확정 항목으로 고정했다.
