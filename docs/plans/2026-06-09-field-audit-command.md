# 현장 PC audit 명령

## PM 판단

V1 마무리 단계에서는 개발자 PC에서 통과한 테스트만으로 충분하지 않다. 사용자가 실제 시연할 PC에서 적재 정합성과 계산 시간이 다시 확인되어야 한다.

검토한 방법:

1. 문서에 현재 개발 PC 측정값만 적는다.
   - 장점: 구현이 없다.
   - 단점: 현장 PC 성능이나 환경 차이를 확인하지 못한다.
2. 앱 안에 audit 버튼을 추가한다.
   - 장점: 비개발자도 누를 수 있다.
   - 단점: V1 실제 작업 화면에 내부 검증 기능이 노출된다.
3. `npm run field:audit` 명령을 추가한다.
   - 장점: 시연 담당자가 프로젝트 폴더에서 한 번 실행해 현장 PC 기준 결과를 확인할 수 있다.
   - 단점: 명령어 실행은 여전히 시연 담당자 역할이 필요하다.

채택: 3번. 현장 작업자 UI를 복잡하게 만들지 않고, 시연 담당자에게 재측정 수단을 제공한다.

## 역할별 검토

- product-manager: V1 시연 준비 과정에서 개발 PC와 현장 PC의 차이를 줄인다.
- business-analyst: 현장 작업자가 직접 개발 검증을 이해할 필요는 없지만, 시연 담당자는 통과/확인 필요 문구를 읽을 수 있어야 한다.
- ui-ux-tester: 앱 화면에는 영향을 주지 않으므로 기존 모바일/태블릿 흐름을 흔들지 않는다.
- code-reviewer: audit 명령은 기존 `runFieldPackingScenarioPerformanceAudit`를 재사용하고 실패 시 exit code 1을 반환해야 한다.
- nextjs-developer: 보고서 문구는 순수 helper로 분리해 단위 테스트를 둔다.

## 수용 기준

- `npm run field:audit` 명령이 존재한다.
- 명령은 현장형 preset 시나리오 audit을 실행한다.
- 통과 시 `Package Tetris 현장 audit 통과`와 요약 수치를 출력한다.
- 안전 실패 또는 계산 지연이 있으면 `Package Tetris 현장 audit 확인 필요`와 확인 대상을 출력하고 종료 코드 1을 반환한다.
- 현장 시연 가이드에 실행 위치와 해석 방법을 반영한다.
