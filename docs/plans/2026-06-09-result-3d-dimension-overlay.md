# 결과 3D 치수 오버레이

## PM 판단

현장 작업자는 3D 적재 결과를 볼 때 박스 위치뿐 아니라 선택 공간의 실제 스케일도 즉시 확인해야 한다. 기존 화면은 결과 툴바에 적재 가능 크기를 표시하지만, 3D 캔버스 안에서는 가로, 깊이, 높이를 바로 읽기 어렵다.

이번 증분은 3D 렌더링 알고리즘을 바꾸지 않고, 캔버스 위에 작은 공간 치수 오버레이를 추가한다.

## 검토한 접근

1. 3D 축 라벨을 Three.js 객체로 렌더링
   - 장점: 장면 안의 축과 함께 움직인다.
   - 단점: 카메라 각도, 가림, 글자 선명도 문제가 생길 수 있다.
2. 캔버스 위 DOM 오버레이
   - 장점: 글자가 선명하고 모바일에서 줄바꿈을 제어하기 쉽다.
   - 단점: 3D 물체에 붙은 라벨은 아니다.
3. 배치 상세표만 보강
   - 장점: 구현이 작다.
   - 단점: 현장 작업자가 3D를 보면서 스케일을 바로 읽기 어렵다.

채택: 2번.

## 적용 내용

- `Result3DCanvas` 안에 `3D 공간 치수` 오버레이를 추가한다.
- 오버레이는 `가로`, `깊이`, `높이`를 mm 단위로 표시한다.
- 오버레이는 `pointer-events: none`으로 3D 클릭, 회전, 확대 조작을 막지 않는다.
- 모바일에서는 한 컬럼으로 접혀 390px 폭에서도 캔버스를 밀지 않는다.

## 역할별 검토

- product-manager: V1 시연에서 3D 결과 해석 시간을 줄이는 작은 개선이다.
- business-analyst: IT 지식이 없는 현장 작업자도 3D 결과의 공간 크기를 바로 확인할 수 있다.
- ui-designer: 오버레이는 캔버스 안쪽에 작게 두고, 조작 버튼이나 툴팁보다 낮은 정보 밀도로 유지한다.
- ui-ux-tester: 모바일에서 오버레이가 한 컬럼으로 접히고 가로 넘침이 없어야 한다.
- code-reviewer: Three.js scene 객체를 건드리지 않고 DOM/CSS만 보강해 렌더링 회귀 위험을 낮춘다.
- nextjs-developer: 기존 `bounds` prop을 사용하므로 저장 스키마와 엔진 출력은 변경하지 않는다.

## 수용 기준

- 3D 결과 영역은 공간 가로, 깊이, 높이를 mm 단위로 표시한다.
- 오버레이는 3D 캔버스 클릭과 포인터 조작을 방해하지 않는다.
- 모바일 390px, 태블릿 768px, 데스크톱 1280px에서 horizontal overflow가 없다.
- 기존 3D 렌더링, 크게 보기, 위/앞/옆 fallback은 유지된다.
