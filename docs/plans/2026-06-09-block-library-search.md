# 저장된 박스 검색

## Product Manager Decision

현장 시연과 실제 사용에서는 저장된 박스가 누적된다. 목록이 길어지면 작업자가 같은 박스를 다시 쓰기 위해 스크롤을 반복해야 하고, 비슷한 박스명을 잘못 선택할 가능성이 커진다.

검토한 방법:

1. 최근 저장순 정렬만 유지
   - 구현은 작지만 이름이 비슷한 박스가 많을 때 찾기 어렵다.
2. 저장된 박스 검색 추가
   - 이름, 치수, 깨짐주의 여부로 필터링할 수 있어 현장 작업자가 눈에 보이는 정보 그대로 찾을 수 있다.
3. 카테고리/태그 체계 추가
   - 장기적으로 유용하지만 V1에는 입력 부담과 관리 비용이 크다.

채택: 2번. 저장 데이터는 변경하지 않고 화면 표시 목록만 필터링한다.

## Role Review

- business-analyst: 사용자는 작업마다 같은 박스를 반복 재사용한다. 검색은 `저장된 박스`와 `이번 작업`의 분리를 유지하면서 재사용 시간을 줄인다.
- ui-ux-tester: 360px 모바일에서도 입력은 48px 터치 타깃을 유지하고, 검색 결과가 없을 때 다음 행동 문구를 제공해야 한다.
- code-reviewer: 필터링은 순수 유틸 `searchBlockTemplates`로 고정하고, 저장된 템플릿 배열 자체는 변경하지 않는다.
- nextjs-developer: React local state로 검색어를 관리한다. IndexedDB schema나 백업 파일 형식은 바꾸지 않는다.

## Acceptance Criteria

- `저장된 박스` 영역에 검색 입력이 보인다.
- 이름, 치수 숫자, `깨짐주의`, `일반` 문구로 저장된 박스를 찾을 수 있다.
- 검색어가 비어 있으면 전체 저장된 박스를 보여준다.
- 검색 결과가 없으면 다른 이름이나 치수로 다시 찾으라는 문구를 보여준다.
- 모바일에서 가로 넘침 없이 검색 입력이 48px 이상 터치 타깃을 유지한다.
