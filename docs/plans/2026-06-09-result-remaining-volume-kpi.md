# Package Tetris V1 Result Remaining Volume KPI

**Goal:** 결과 상단 요약에서 현장 작업자가 전체 사용 공간 기준으로 남은 부피를 바로 확인할 수 있게 한다.

## Context

V1 기획서는 결과 화면에서 `사용 공간 수`, `평균 적재율`, `남은 부피`, `미적재 박스 수`, `실행 시간`을 확인한다고 정의한다. 현재 화면은 사용 공간, 평균 적재율, 미적재, 대상 공간을 보여주지만 `남은 부피`가 빠져 있다.

## Approaches

1. `ResultSummary`에 `remainingVolumeM3`를 저장한다.
   - 장점: 계산 결과 객체만 보면 값을 알 수 있다.
   - 단점: 백업 JSON과 기존 IndexedDB 작업본의 호환 범위를 넓힌다.

2. 화면에서 결과 공간과 적재 가능 크기로 파생 계산한다.
   - 장점: 저장 스키마를 바꾸지 않고 V1 화면 누락만 보완한다.
   - 단점: 계산 유틸과 UI 계약 테스트가 필요하다.

3. 공간별 남은 부피 상세 표를 추가한다.
   - 장점: 더 자세하다.
   - 단점: 결과 화면이 길어지고 모바일 밀도가 나빠진다.

**Decision:** 2번을 적용한다. V1은 서버 없는 프론트 단독과 백업 호환성이 중요하므로 파생 계산이 가장 안전하다.

## Role Review

- **product-manager:** 기획서의 V1 결과 요약 누락을 작게 보완한다. 저장 구조와 엔진 출력은 변경하지 않는다.
- **business-analyst:** 현장 작업자는 남은 공간을 부피 기준으로 빠르게 보고 추가 적재 가능성을 판단한다.
- **ui-designer:** 결과 상단 KPI는 메인 영역이므로 기존 요약 타일에 한 칸을 추가하고, 화면이 좁으면 자연스럽게 2열/1열로 접는다.
- **ui-ux-tester:** 360px, 390px, 768px, 1280px에서 결과 요약 타일이 가로 넘침 없이 표시되어야 한다.
- **nextjs-developer:** 순수 유틸로 계산과 포맷을 분리하고, 컴포넌트는 값 표시만 담당한다.

## Acceptance Criteria

- 결과가 없으면 `남은 부피`는 `-`로 표시된다.
- 결과가 있으면 사용 공간 전체의 적재 가능 부피에서 적재된 박스 부피를 뺀 값을 `m³`로 표시한다.
- 계산 값은 음수가 되지 않는다.
- `Intl.NumberFormat` 기반으로 최대 3자리 소수까지 현장 화면에 표시한다.
- 결과 상단 타일은 5개가 되어도 모바일에서 가로 overflow를 만들지 않는다.
- `npm test`, `npx tsc --noEmit`, `npm run build`, 브라우저 검증을 통과한다.

## References

- MDN `Intl.NumberFormat`: `maximumFractionDigits`로 소수 표시 자릿수를 제한할 수 있다.
- W3C WCAG Target Size: 모바일 터치 환경에서는 주요 조작 대상이 충분한 터치 크기를 가져야 한다.
