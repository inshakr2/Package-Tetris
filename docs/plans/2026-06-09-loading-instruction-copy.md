# Loading Instruction Copy Plan

> V2 기준: 이 문서는 2026-06-09 V1 구현 이력입니다. V2에서는 배치상세/쌓는순서/작업지시서 산출물을 제거하고 3D와 공간 확인, 미적재/경고, 백업 상태 중심으로 대체했습니다.

## Context

이번 증분은 결과 영역의 `쌓는 순서`를 현장 작업자가 휴대폰이나 메신저로 바로 전달할 수 있게 만드는 작은 UI 개선이다. V1은 서버 없이 프론트 단독으로 운영되므로 공유 기능은 브라우저 클립보드 복사로 제한한다.

참고 자료:
- MDN Clipboard `writeText()`: 텍스트를 시스템 클립보드에 쓰고 Promise로 완료 여부를 돌려주며, 보안 컨텍스트에서 동작한다.
- W3C WCAG 2.2 Target Size Minimum: 포인터 입력 대상은 최소 24x24 CSS px 기준을 충족해야 한다. 현장 터치 UX는 프로젝트 기준인 48px 이상으로 더 보수적으로 잡는다.

## Compared Approaches

1. **복사 버튼만 추가**
   - 장점: 구현 범위가 작고 오프라인/프론트 단독 범위에 맞다.
   - 단점: 클립보드 권한 또는 브라우저 지원이 막히면 사용자가 실패 사유를 알아야 한다.

2. **공유 API까지 추가**
   - 장점: 모바일 공유 시트로 전달성이 좋다.
   - 단점: 브라우저 지원과 플랫폼 차이가 있고 이번 증분보다 범위가 커진다.

3. **PDF/인쇄 지시서 생성**
   - 장점: 현장 문서화에 좋다.
   - 단점: 레이아웃 검증과 출력 UX가 필요해 별도 기능 단위가 더 안전하다.

선택: 1번. 현장 문장 복사만 제공하고, 실패하면 권한/브라우저 상태를 안내한다. `navigator.clipboard.writeText`를 우선 사용하되, API가 없는 현장 브라우저에서는 임시 textarea 기반 복사 폴백을 사용한다. 공유 API와 인쇄 지시서는 후속 증분으로 남긴다.

## Role Review

- product-manager: 현재 결과 패널의 핵심 흐름을 유지하면서 전달성을 높이는 작은 증분이므로 진행한다.
- business-analyst: 작업자가 IT 용어 없이 `1층`, `2층` 순서 문장을 그대로 전달할 수 있어 현장 언어 기준에 맞다.
- ui-designer: 버튼은 결과 패널 헤더 우측에 두고 모바일에서는 한 줄 전체 폭으로 접어 터치 실수를 줄인다.
- ui-ux-tester: 360px/390px에서 가로 넘침이 없어야 하고, 복사 버튼은 48px 이상 터치 영역을 유지해야 한다.
- nextjs-developer: 순수 유틸에서 복사용 텍스트를 만들고, UI는 클라이언트 상태와 클립보드 유틸만 사용한다.
- code-reviewer: SSR 의존, 권한 실패, 빈 결과 복사 방지, 불필요한 새 패키지 추가를 확인한다.

## Acceptance Criteria

- `쌓는 순서`가 있으면 `작업 순서 복사` 버튼이 표시된다.
- 버튼을 누르면 공간명과 층별 지시가 여러 줄 텍스트로 복사된다.
- 복사 성공/실패 상태가 화면에 문장으로 표시된다.
- 적재 지시가 없으면 복사 버튼은 비활성화된다.
- 모바일에서 버튼과 상태 문구가 패널 밖으로 넘치지 않는다.
