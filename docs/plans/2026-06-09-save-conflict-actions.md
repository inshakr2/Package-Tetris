# Save Conflict Banner Inline CTA Plan

## 목적

저장 충돌로 화면이 읽기 전용이 되었을 때, 현장 작업자가 본문 아래나 별도 패널로 내려가지 않고 상단 banner 안에서 바로 다음 행동을 고를 수 있게 한다.

## 검토한 3가지 접근안

1. 상단 banner에 inline CTA 2개를 바로 배치한다.
   - 장점: 충돌 원인과 다음 행동이 한 자리에서 끝난다.
   - 장점: 읽기 전용 상태에서도 클릭 가능한 영역이 명확하다.
   - 단점: 배너 밀도가 약간 높아진다.

2. 상단 banner는 유지하고 저장 상태 패널 열기 버튼만 강조한다.
   - 장점: 기존 구조 변경이 적다.
   - 단점: 현장 작업자가 한 번 더 패널을 열어야 하므로 충돌 대응이 느리다.
   - 단점: 읽기 전용 상태의 다음 행동이 banner 안에서 바로 보이지 않는다.

3. 충돌 전용 modal 또는 sheet를 추가한다.
   - 장점: 정보와 행동을 크게 보여줄 수 있다.
   - 단점: 흐름이 무거워지고, 별도 모달 상태 관리가 늘어난다.
   - 단점: 이번 요구사항의 "별도 모달 없이 기존 banner 확장"과 맞지 않는다.

## 채택안

1번을 채택한다. 기존 `.workspace-readonly-banner`를 행동 배너로 확장하고, `최신본 불러오기`와 `현재 화면 백업` CTA를 바로 제공한다.

## 채택 근거

- 충돌 상태의 핵심 질문은 "지금 무엇을 해야 하나"다.
- 저장 상태 패널까지 이동시키는 방식보다 inline CTA가 현장 터치 환경에서 더 빠르다.
- 기존 읽기 전용 잠금 구조를 유지하면서도 banner만 예외적으로 클릭 가능하게 두면 범위가 가장 좁다.
- copy를 순수 helper로 분리하면 `remote`와 `storage` 충돌 문구 차이를 테스트로 고정할 수 있다.
- banner 세부 정보는 개발자 용어인 revision 대신 현장 작업자가 이해하기 쉬운 작업본 번호로 표현한다.

## 수용조건

- 상단 banner에 행동 중심 제목과 설명이 보인다.
- primary CTA는 `최신본 불러오기`, secondary CTA는 `현재 화면 백업`이다.
- 읽기 전용 pointer 차단 상태에서도 banner CTA는 눌려야 한다.
- 모바일 360px, 390px에서 가로 스크롤 없이 문구와 버튼이 줄바꿈된다.
- 버튼 높이는 최소 48px 수준을 유지하고 텍스트가 잘리지 않는다.
- copy helper가 있다면 source별 문구와 작업본 번호 detail을 테스트로 검증한다.
- banner 본문에는 `revision` 같은 개발자 용어가 직접 노출되지 않는다.
