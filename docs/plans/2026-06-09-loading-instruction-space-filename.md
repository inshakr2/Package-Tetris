# Loading Instruction Space Filename

## Goal

`작업 지시서 저장` 파일명에 실제 공간명을 반영해, 현장 작업자가 다운로드 폴더에서 어떤 적재 공간의 지시서인지 바로 찾을 수 있게 한다.

## Context

- 직전 증분에서 `쌓는 순서` 헤더와 복사/저장 본문에는 `공간명 · Space N`을 포함했다.
- 하지만 다운로드 파일명은 여전히 `my-tetris-space-1-loading-YYYY-MM-DD.txt` 형태라, 여러 공간 결과를 저장하면 파일 목록만 보고 구분하기 어렵다.
- MDN은 `HTMLAnchorElement.download` 값이 기본 파일명 역할을 하지만 OS에서 유효하지 않은 이름이면 브라우저가 조정할 수 있다고 설명한다: https://developer.mozilla.org/en-US/docs/Web/API/HTMLAnchorElement/download
- Microsoft Windows 파일명 규칙은 `< > : " / \ | ? *`, 제어문자, 일부 예약 이름을 피해야 한다고 안내한다: https://learn.microsoft.com/windows/win32/fileio/naming-a-file

## Product Review

- product-manager: 작업 지시서가 화면 밖 다운로드 폴더로 이동한 뒤에도 식별 가능해야 하므로, 본문뿐 아니라 파일명도 공간명을 가져야 한다.
- business-analyst: 현장 사용자는 파일 시스템 규칙을 모를 수 있으므로, 시스템이 공간명을 안전한 파일명으로 바꿔줘야 한다.
- ui-designer: 완료 문구 영역은 이미 긴 파일명 줄바꿈을 허용하므로 별도 UI 확대 없이 파일명만 개선한다.
- ui-ux-tester: 커스텀 공간명에 공백, 한글, 특수문자, 긴 문구가 있어도 모바일에서 완료 문구가 패널 밖으로 넘치지 않아야 한다.
- nextjs-developer: 파일명 정규화는 순수 유틸로 만들고, 컴포넌트는 선택된 결과 공간명만 전달한다.

## Compared Approaches

1. 기존 `space-N` 파일명을 유지한다.
   - 장점: 변경 없음.
   - 단점: 다운로드 폴더에서 팔레트/트럭/컨테이너 지시서를 구분하기 어렵다.
2. 파일명에 `공간명 slug + space-N`을 포함한다.
   - 장점: 현장 식별성이 높고 기존 저장 흐름을 유지한다.
   - 단점: 공간명 정규화 규칙이 필요하다.
3. 파일명은 유지하고 완료 문구에만 공간명을 추가한다.
   - 장점: 파일명 호환성 위험이 거의 없다.
   - 단점: 다운로드 폴더나 메신저 첨부 목록에서는 여전히 구분하기 어렵다.

선택: 2번. 파일명은 한글을 보존하는 안전 slug로 정규화하고, 공간명이 비어 있거나 정규화 후 남는 문자가 없으면 기존 `space-N` fallback을 유지한다.

## Acceptance Criteria

- 공간명이 있으면 파일명에 안전한 공간명 slug와 `space-N`이 함께 포함된다.
- 한글은 보존하고 공백, `/`, `?`, `*` 같은 문자는 안전하게 정규화된다.
- 공간명이 없거나 정규화 후 비어 있으면 기존 `space-N` 파일명을 사용한다.
- 저장 성공 문구는 실제 생성된 파일명을 그대로 보여준다.
- 모바일 360px/390px에서 완료 문구와 버튼이 가로로 넘치지 않는다.
- 전체 테스트, 타입체크, 빌드, 브라우저 검증 통과 후에만 커밋/푸시한다.
