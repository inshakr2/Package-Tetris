# Loading Instruction Space Name Context

> V2 기준: 이 문서는 2026-06-09 V1 구현 이력입니다. V2에서는 배치상세/쌓는순서/작업지시서 산출물을 제거하고 3D와 공간 확인, 미적재/경고, 백업 상태 중심으로 대체했습니다.

## Goal

`쌓는 순서`, 복사 텍스트, 작업 지시서 저장 내용에 실제 대상 공간명을 함께 표시해 현장 작업자가 `Space 1`만 보고 공간을 혼동하지 않게 한다.

## Product Review

- product-manager: 여러 공간 인스턴스가 생기면 `Space 1`은 순서 구분에는 충분하지만, 팔레트/트럭/컨테이너 같은 대상 공간 확인에는 부족하다.
- business-analyst: 현장 사용자는 결과 화면 밖에서 작업 지시서를 볼 수 있으므로, 지시서 자체에 대상 공간명이 들어가야 한다.
- ui-designer: 패널 제목과 버튼 영역은 유지하고, 보조 문구와 텍스트 헤더만 확장해 화면 밀도를 크게 늘리지 않는다.
- ui-ux-tester: 긴 커스텀 공간명은 기존 줄바꿈 규칙으로 패널 밖으로 넘치지 않아야 한다.
- nextjs-developer: 라벨 생성은 순수 함수로 분리해 공백 정규화와 fallback을 테스트한다.

## Compared Approaches

1. 기존 `Space N`만 유지한다.
   - 장점: 변경 없음.
   - 단점: 저장된 지시서를 화면 밖에서 볼 때 어떤 공간 기준인지 헷갈릴 수 있다.
2. `공간명 · Space N` 라벨을 UI와 작업 지시서 헤더에 사용한다.
   - 장점: 현장 식별성이 높고 변경 범위가 작다.
   - 단점: 공간명이 길면 한 줄 문구가 늘어난다.
3. 모든 층별 행마다 공간명을 반복 표시한다.
   - 장점: 각 행만 복사해도 맥락이 남는다.
   - 단점: 모바일에서 정보가 반복되고 읽기 피로가 커진다.

선택: 2번. 공간명은 헤더와 보조 문구에 한 번만 표시하고, 층별 지시는 지금처럼 간결하게 유지한다.

## Acceptance Criteria

- 선택 공간명이 있으면 `공간명 · Space N` 라벨을 만든다.
- 공간명이 없으면 기존처럼 `Space N` 또는 `선택한 공간`을 사용한다.
- 복사/저장되는 작업 지시서 첫 줄에 대상 공간명이 포함된다.
- 결과 화면의 `쌓는 순서` 보조 문구도 같은 공간명을 보여준다.
- 긴 공간명은 모바일에서 패널 밖으로 넘치지 않는다.
