# Packing Engine Stability Design

**Goal:** 적재 결과에 공중에 떠 있는 박스가 나오지 않도록 계산 엔진 단계에서 안정 지지, 3D 충돌, 공간 경계, 깨짐주의 적층 정책을 보장한다.

## Product Manager Decision

채택 접근안은 `엔진 배치 후보를 지지면 기반으로 재구성`이다.

1. 렌더러 보정
   - 3D 화면에서만 떠 있는 박스를 내려 보일 수 있지만 실제 계산 결과가 거짓이 된다.
2. 결과 검증만 추가
   - 잘못된 결과를 잡을 수는 있지만 현장 작업자는 여전히 사용할 수 없는 결과를 받을 수 있다.
3. 지지면 기반 배치 엔진
   - 계산 결과 자체가 현장 적재 규칙을 지키므로 제품 신뢰성을 직접 개선한다.

## Scope

- `packing-engine`은 배치 후보를 `x/y/z` 경계 후보와 회전 후보 조합으로 만들고, 가장 낮은 `z`, 앞쪽 `y`, 왼쪽 `x` 순서로 선택한다.
- 후보는 공간 경계 안에 있어야 한다.
- 후보는 기존 박스와 3D 충돌하면 안 된다.
- 후보의 `zMm`이 0보다 크면 같은 높이의 하부 박스들이 후보 박스 바닥 전체 면적을 지지해야 한다.
- 일반 박스는 깨짐주의 박스 위에 올라갈 수 없다.
- 깨짐주의 박스는 깨짐주의 박스 위에 올라갈 수 있다.
- `chain-simulation`과 `packing-engine`은 같은 안정성 규칙을 공유한다.
- `chain-simulation`은 하드코딩된 깨짐주의 정책을 쓰지 않고 현재 작업의 정책을 입력으로 받는다.
- 실행 전 확인의 공간 수는 `부피 기준 최소 공간 수`로 표현한다. 이는 참고 최소값이며, 실제 결과는 받쳐 주는 바닥과 쌓는 규칙 때문에 늘어날 수 있음을 함께 안내한다.
- 실제 사용 공간 수가 부피 기준 최소 공간 수보다 커지면 결과 상단에 현장 안내를 표시한다.

## Acceptance Criteria

- 기존 cursor 기반 엔진에서 공중 배치가 가능했던 케이스가 이제 바닥의 남은 공간 또는 새 공간으로 배치된다.
- 모든 `PackedBlock`은 공간 적재 가능 크기 안에 있다.
- 같은 공간 안의 `PackedBlock`끼리 3D overlap이 없다.
- `zMm > 0`인 모든 `PackedBlock`은 하부 박스 지지면이 바닥 면적 전체를 덮는다.
- 일반 박스는 깨짐주의 하부 박스 위에 배치되지 않는다.
- 체이닝 시뮬레이션은 현재 작업의 깨짐주의 적층 정책을 그대로 따른다.
- 공간이 늘어난 결과는 `부피로는 남아 보여도 안전하게 받칠 바닥이 부족해 공간이 나뉘었습니다.`처럼 현장 작업자가 이해할 수 있는 문구로 설명한다.
- 미적재가 발생하면 `안전하게 올릴 자리가 없음`과 다음 행동을 포함한 문구로 설명한다.
- `npm test`, `npx tsc --noEmit`, `npm run build`가 통과한다.

## Role Review Loop

- `nextjs-developer`: TDD로 실패 테스트를 추가한 뒤 공통 placement 유틸과 엔진을 구현한다.
- `code-reviewer`: 공통화, 결정론, edge case, 기존 chain simulation 회귀 여부를 검토한다.
- `business-analyst`: 현장 적재 규칙과 미적재/경고 문구가 요구사항을 만족하는지 확인한다.
- `ui-ux-tester`: 결과 생성 후 3D/2D 화면에서 비정상 좌표가 보이지 않는지, 경고가 이해 가능한지 확인한다.
