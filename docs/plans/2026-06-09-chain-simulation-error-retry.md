# 추가 박스 시뮬레이션 오류 후 재계산

## PM 판단

추가 박스 시뮬레이션에서 기준 결과가 일시적으로 검증 실패하거나 계산 예외가 발생하면, 현장 작업자는 같은 박스로 다시 계산해 보고 싶을 수 있다. 기존 화면은 오류 상태에서 `최대 적재 계산` 버튼까지 비활성화되어, 사용자가 같은 박스를 다시 누르거나 결과를 다시 생성해야 한다고 추론해야 했다.

이번 증분은 계산 엔진을 바꾸지 않고 오류 후 복구 행동만 명확히 한다.

## 검토한 접근

1. 오류 상태에서는 결과 다시 생성만 제공한다.
   - 장점: 기존 구조를 유지한다.
   - 단점: 일시 오류나 같은 조건 재시도에 불필요하게 큰 행동을 요구한다.
2. 오류 상태에서도 선택한 박스 기준 `다시 계산`을 허용한다.
   - 장점: 가장 가까운 복구 행동이 버튼으로 바로 보인다.
   - 단점: 기준 결과 자체가 안전하지 않으면 같은 오류가 다시 표시될 수 있다.
3. 오류 전용 복구 패널을 추가한다.
   - 장점: 안내는 더 자세하다.
   - 단점: V1 마무리 범위에 비해 화면이 길어진다.

채택: 2번. 같은 오류가 반복되더라도 상태 문구가 유지되고, 사용자는 필요할 때 `결과 다시 생성`으로 넘어갈 수 있다.

## 적용 범위

- `ChainSimulationPanel`에서 오류 상태도 선택 박스가 있으면 계산 버튼을 활성화한다.
- 오류 상태의 계산 버튼 문구는 `다시 계산`으로 표시한다.
- 엔진, 저장 구조, 체이닝 이력 반영 방식은 변경하지 않는다.

## 역할별 검토

- business-analyst: 실패 후 복구 행동은 실패 문구 근처에 있어야 한다.
- ui-ux-tester: 오류 상태에서 버튼이 모두 막힌 듯 보이지 않아야 한다.
- code-reviewer: 오류 원인 판단 로직은 유지하고, 재시도 버튼만 열어 회귀 범위를 줄인다.
- nextjs-developer: 기존 `onCalculate` 흐름을 재사용하므로 추가 상태나 API가 필요 없다.

## 수용 기준

- 오류 상태에서 선택 박스가 있으면 `다시 계산` 버튼이 활성화된다.
- 계산 중에는 기존처럼 버튼이 비활성화된다.
- 오류 상태의 `결과 다시 생성` 보조 행동은 유지된다.
- `npm test`, `npx tsc --noEmit`, `npm run build`, 브라우저 검증을 통과한 뒤 커밋/푸시한다.
