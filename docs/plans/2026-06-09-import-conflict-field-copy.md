# 백업 파일 가져오기 충돌 문구 개선

## PM 판단

V1 시연에서 백업 파일 가져오기는 서버 없는 구조의 핵심 복구 경로다. 기존 패널은 `same-file-revision-conflict` 같은 내부 충돌 코드를 그대로 보여줘 IT 도메인 지식이 없는 현장 작업자가 의미를 파악하기 어렵다.

이번 증분은 가져오기 로직과 저장 스키마를 바꾸지 않고, 충돌 안내 문구와 버튼 라벨만 현장 언어로 정리한다.

## 검토한 접근

1. 기존 문구 유지
   - 장점: 변경 위험이 없다.
   - 단점: 현장 작업자가 내부 코드명을 보고 작업 선택을 망설일 수 있다.
2. 충돌 처리 전체를 모달로 재설계
   - 장점: 선택 집중도가 높다.
   - 단점: 현재 레이아웃과 저장 충돌 UX까지 함께 건드려 범위가 커진다.
3. 기존 패널 유지 + copy helper 분리
   - 장점: 기능 흐름을 유지하면서 문구 품질과 테스트 가능성을 높인다.
   - 단점: 화면 구조 자체의 집중도 개선은 제한적이다.

채택: 3번.

## 적용 내용

- `getImportConflictCopy` helper를 추가해 충돌 종류별 제목, 설명, 백업 안내, 버튼 라벨을 반환한다.
- 백업 파일 가져오기 패널은 더 이상 내부 충돌 kind를 직접 표시하지 않는다.
- `대체` 버튼은 `가져온 파일로 대체`로 바꿔 행동 결과를 명확히 한다.
- import 충돌 버튼 그룹은 모바일에서 48px 이상 터치 타깃과 한 컬럼 접힘을 유지한다.

## 역할별 검토

- product-manager: V1의 백업/복구 신뢰도를 높이는 작은 증분이다.
- business-analyst: 사용자에게 필요한 것은 충돌 코드가 아니라 현재 화면과 가져온 파일 중 무엇을 사용할지 선택하는 안내다.
- ui-designer: 기존 panel을 유지하되 문구와 버튼 의미를 직접화해 화면 복잡도를 늘리지 않는다.
- ui-ux-tester: 360px 모바일에서 버튼이 잘리지 않고, `가져온 파일로 대체` 의미가 명확해야 한다.
- code-reviewer: copy helper를 순수 함수로 분리해 충돌 종류별 문구 회귀를 테스트로 고정한다.
- nextjs-developer: import/export 스키마와 IndexedDB 저장 흐름은 변경하지 않는다.

## 수용 기준

- 백업 가져오기 충돌 패널에 `same-file`, `revision`, `conflict`, `different-file` 같은 내부 코드명이 표시되지 않는다.
- 같은 작업본의 다른 저장본은 `같은 작업의 다른 저장본을 가져왔습니다.`로 안내한다.
- 다른 작업본 백업은 `다른 작업의 백업 파일입니다.`로 안내한다.
- 사용자는 현재 화면 유지, 가져온 파일로 대체, 복사본으로 열기, 가져오기 취소를 구분할 수 있다.
- 모바일 import 충돌 버튼은 48px 이상 터치 타깃과 전체 폭 접힘을 유지한다.
- `npm test`, `npx tsc --noEmit`, `npm run build`, 브라우저 검증이 통과한다.
