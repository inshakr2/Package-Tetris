# 결과 CTA 접근성 이름 정리

## PM 판단

브라우저 시연 중 `결과 만들기` 버튼이 실행 전 확인 카드, 결과 대기 화면, 결과 작업 패널, 모바일 하단 고정 액션에 반복 노출되는 것이 확인됐다. 시각적으로는 위치가 달라 큰 문제는 아니지만, 자동화 검증과 보조기기에서는 같은 이름의 버튼이 여러 개로 잡혀 사용자가 현재 맥락을 구분하기 어렵다.

검토한 방법:

1. visible text를 모두 다르게 바꾸기
   - 장점: 화면만 봐도 버튼 위치가 더 명확하다.
   - 단점: 현장 사용자가 익숙하게 보는 핵심 CTA 문구가 길어지고 화면 밀도가 올라간다.
2. `aria-label`만 위치별로 구분하기
   - 장점: 보이는 UI는 그대로 유지하면서 보조기기와 테스트 자동화의 모호성을 줄인다.
   - 단점: 시각 사용자에게는 변화가 거의 보이지 않는다.
3. 중복 CTA 자체를 줄이기
   - 장점: 화면이 단순해진다.
   - 단점: 모바일 고정 액션, 결과 근처 액션처럼 이미 의도된 진입 경로를 줄일 수 있다.

채택: 2번. V1 시연 화면의 visible text는 유지하고, 위치별 접근성 이름만 구분한다.

## Role Review

- business-analyst: 현장 작업자는 `결과 만들기`라는 짧은 문구를 반복해서 보는 편이 이해하기 쉽다.
- ui-designer: 시각 레이아웃과 버튼 밀도는 바꾸지 않고, 숨은 접근성 이름만 보강한다.
- ui-ux-tester: 390px 모바일에서 하단 고정 액션과 본문 CTA가 함께 존재해도 보조기기/자동화가 구분할 수 있어야 한다.
- code-reviewer: 기능 흐름을 바꾸지 않고 순수 helper와 source-level test로 회귀를 막는다.
- nextjs-developer: `mobile-sticky-action` helper와 `tetris-workspace-app` aria-label만 좁게 수정한다.

## Acceptance Criteria

- 실행 전 확인 카드의 CTA는 `실행 전 확인에서 결과 만들기`로 구분된다.
- 결과 대기 화면 CTA는 `결과 대기 화면에서 결과 만들기`로 구분된다.
- 결과 작업 패널 CTA는 결과가 없을 때 `결과 작업에서 결과 만들기`, 결과가 있을 때 `현재 입력으로 다시 계산`으로 구분된다.
- 모바일 하단 고정 CTA는 `모바일 하단에서 ...` 위치 정보를 포함한다.
- 보이는 버튼 문구 `결과 만들기`, `결과 다시 만들기`, `백업 만들기`는 기존 현장 문구를 유지한다.
