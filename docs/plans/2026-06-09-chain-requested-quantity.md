# 추가 박스 시뮬레이션 수량 직접 지정

## PM 판단

V1 기획서는 추가 박스 시뮬레이션에서 `최대 적재 계산`을 기본 액션으로 두되, 현장 작업자가 원하는 수량만 시험할 수 있는 보조 액션도 제공한다고 정리한다. 현재 화면은 최대 적재만 지원해 `3개만 더 넣으면 가능한가?` 같은 현장 질문을 바로 확인하기 어렵다.

이번 증분은 새 공간 생성이나 수동 배치 편집 없이, 기존 결과 공간 안에서 선택한 박스를 요청 수량까지만 추가 배치한다.

## 검토한 접근

1. 최대 적재 계산만 유지하고 결과 문구를 보강한다.
   - 장점: 구현 변경이 없다.
   - 단점: 원하는 수량을 시험하려는 현장 질문을 해결하지 못한다.
2. 수량 입력과 `지정 수량 계산` 버튼을 추가한다.
   - 장점: 기존 엔진 흐름을 재사용하면서 원하는 수량 확인을 바로 지원한다.
   - 단점: 패널에 입력 컨트롤이 하나 늘어난다.
3. 여러 수량 결과를 한 번에 비교하는 UI를 만든다.
   - 장점: 분석 기능은 강하다.
   - 단점: V1 현장 사용성 대비 화면과 검증 범위가 커진다.

채택: 2번. V1에서는 한 번에 하나의 수량만 시험하는 방식이 가장 작고 명확하다.

## 적용 범위

- `runChainSimulationV0`에 `requestedQuantity`를 선택 입력으로 추가한다.
- 값이 없으면 기존처럼 최대 적재량까지 계산한다.
- 값이 있으면 최대 적재 가능량과 요청 수량 중 작은 값까지만 추가한다.
- 요청 수량이 1개 미만이면 계산하지 않고 현장 문구로 안내한다.
- 체이닝 패널에 `원하는 추가 수량(개)` 입력과 `지정 수량 계산` 버튼을 추가한다.

## 역할별 검토

- business-analyst: 현장 작업자는 최대치보다 실제 남은 주문 수량이나 추가 출고 수량을 확인하는 경우가 많다.
- ui-designer: 기존 체이닝 패널 안에 한 줄 컨트롤로 넣고, 태블릿 이하에서는 한 컬럼으로 접어 가독성을 유지한다.
- ui-ux-tester: 모바일 360px/390px에서 입력과 버튼이 48px 터치 타깃을 유지해야 한다.
- code-reviewer: 최대 적재 기존 동작은 그대로 유지하고, 요청 수량 제한은 순수 엔진 테스트로 고정한다.
- nextjs-developer: 서버, 저장 스키마, 체이닝 이력 구조는 바꾸지 않는다.

## 수용 기준

- 최대 적재 계산은 기존 결과를 유지한다.
- 지정 수량 계산은 요청 수량 이하만 추가한다.
- 요청 수량 0 이하 입력은 추가 계산을 하지 않는다.
- 모바일/태블릿에서 수량 입력과 버튼이 가로 넘침 없이 표시된다.
- `npm test`, `npx tsc --noEmit`, `npm run build`, 브라우저 검증을 통과한 뒤 커밋/푸시한다.
