# Loading Instruction Calculated Time

> V2 기준: 이 문서는 2026-06-09 V1 구현 이력입니다. V2에서는 배치상세/쌓는순서/작업지시서 산출물을 제거하고 3D와 공간 확인, 미적재/경고, 백업 상태 중심으로 대체했습니다.

## Goal

복사/저장되는 `작업 지시서` 본문에 계산 시각을 포함해, 현장 작업자가 화면 밖에서 지시서를 봐도 어느 계산 결과 기준인지 확인할 수 있게 한다.

## Context

- 결과 화면에는 이미 `계산 시각`이 표시된다.
- 최근 증분에서 작업 지시서 본문과 파일명에는 공간명이 들어가게 됐다.
- 하지만 복사된 텍스트나 저장된 `.txt` 파일 본문에는 계산 시각이 없어, 여러 번 결과를 만들고 지시서를 전달하면 어떤 결과 기준인지 혼동할 수 있다.
- MDN은 `Intl.DateTimeFormat`이 언어/지역에 맞는 날짜와 시간 포맷을 제공한다고 설명한다: https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl/DateTimeFormat

## Product Review

- product-manager: 지시서가 화면 밖으로 전달되는 순간에는 공간명, 경고, 미적재 수뿐 아니라 계산 기준 시각도 필요하다.
- business-analyst: 현장 작업자는 `방금 만든 지시서인지`, `어제 만든 지시서인지`를 파일 본문만 보고 판단할 수 있어야 한다.
- ui-designer: 기존 패널 레이아웃은 유지하고, 복사/저장 텍스트 첫 줄 아래에 한 줄만 추가해 화면 밀도를 늘리지 않는다.
- ui-ux-tester: 날짜 문구는 길어질 수 있으므로 상태 문구와 지시 리스트의 모바일 줄바꿈 기준을 유지해야 한다.
- nextjs-developer: 텍스트 생성 유틸에 `calculatedAtLabel` 옵션을 추가하고, 컴포넌트는 `latestResult.createdAt`을 현장용 포맷으로 넘긴다.

## Compared Approaches

1. 계산 시각은 결과 화면에만 둔다.
   - 장점: 변경 없음.
   - 단점: 복사/저장된 지시서만 전달되면 기준 결과를 알기 어렵다.
2. 지시서 헤더 아래에 `계산 시각: YYYY. M. D. 오전/오후 HH:MM` 한 줄을 추가한다.
   - 장점: 가장 작은 변경으로 본문 단독 식별성이 높아진다.
   - 단점: 복사 텍스트가 한 줄 길어진다.
3. 모든 층별 지시 행에 계산 시각을 반복한다.
   - 장점: 일부 행만 잘라 전달해도 기준 시각이 남는다.
   - 단점: 반복 정보가 많고 모바일/메신저에서 읽기 피로가 크다.

선택: 2번. 계산 시각은 지시서 헤더 바로 아래에 한 번만 표시하고, 층별 작업 지시는 기존처럼 간결하게 유지한다.

## Acceptance Criteria

- 작업 지시서 텍스트는 계산 시각 라벨이 있으면 헤더 다음 줄에 `계산 시각: ...`을 포함한다.
- 계산 시각 라벨이 없으면 기존 텍스트 구조를 유지한다.
- 컴포넌트는 `latestResult.createdAt`을 작업 지시서 전용 날짜/시간 라벨로 변환해 전달한다.
- 복사/저장 버튼과 완료 문구는 모바일 360px/390px에서 가로로 넘치지 않는다.
- 전체 테스트, 타입체크, 빌드, 브라우저 검증 통과 후에만 커밋/푸시한다.
