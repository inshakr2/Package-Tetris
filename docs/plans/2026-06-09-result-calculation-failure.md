# 2026-06-09 Result Calculation Failure

## Goal

결과 계산 중 예외가 발생했을 때 화면에 원인과 복구 행동을 남긴다. 현장 작업자는 계산 실패 후에도 작업본이 유지되는지, 다시 계산해야 하는지, 입력을 줄여야 하는지 바로 알아야 한다.

## Context

- V1 인수 기준은 계산 실패 시 브라우저 계산 중단, 메모리 부족, 입력 제약 위반 등 사용자에게 유효한 실패 원인을 보여주도록 정의한다.
- 현재 `결과 만들기`는 계산 중 상태와 중복 클릭 방지는 제공하지만, 엔진 예외가 발생하면 UI 안내가 남지 않는다.
- 서버 없는 V1에서는 브라우저 내부 예외를 현장 언어로 분류하고, 같은 화면에서 복구 CTA를 제공하는 방식이 가장 작다.

## Options Reviewed

1. Web Worker로 계산을 분리하고 실패 메시지를 표준화
   - 장점: 장기적으로 중단/성능/메모리 대응이 좋다.
   - 단점: V1 마무리 범위보다 크고 기존 엔진 호출 구조를 크게 바꾼다.
2. 실행 전 검토만 강화
   - 장점: 입력 오류를 사전에 줄일 수 있다.
   - 단점: 브라우저 중단이나 런타임 메모리 부족을 설명하지 못한다.
3. 현재 계산 흐름의 예외를 잡아 실패 배너와 복구 CTA 제공
   - 장점: 기존 구조를 유지하면서 수용 기준의 실패 원인 안내를 닫을 수 있다.
   - 단점: 실패 분류는 브라우저 오류 문자열에 일부 의존한다.

## Decision

3번을 선택한다. `createResultCalculationFailure` 순수 헬퍼로 브라우저 중단, 메모리 부족, 입력 제약, 알 수 없는 오류를 현장 문구로 분류하고, 결과 영역에 `계산 실패` 배너와 `입력 수정`, `다시 계산` CTA를 표시한다.

## Role Review

- product-manager: 수용 기준 12번을 작은 범위로 닫고, 실패 후 재시도 경로를 명확히 한다.
- business-analyst: 현장 작업자는 실패 원인보다 다음 행동을 먼저 알아야 하므로, 제목과 행동 힌트를 분리한다.
- ui-designer: 실패 배너는 결과 영역 상단에 두고, CTA는 48px 이상 터치 타깃으로 제공한다.
- ui-ux-tester: 모바일에서 실패 문구와 버튼이 가로로 넘치지 않고, `입력 수정`과 `다시 계산`이 모두 보여야 한다.
- code-reviewer: 엔진 예외 문자열을 그대로 노출하지 않고 순수 헬퍼 테스트로 분류 문구를 고정한다.

## Acceptance

- 계산 중 예외가 발생하면 `계산 실패` 배너가 결과 영역에 표시된다.
- 배너는 실패 제목, 설명, 행동 힌트, `입력 수정`, `다시 계산`을 포함한다.
- 실패 배너는 모바일에서 한 컬럼 CTA로 접힌다.
- 정상 계산 성공 시 이전 실패 안내는 사라진다.
