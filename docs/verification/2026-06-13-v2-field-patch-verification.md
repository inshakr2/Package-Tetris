# Package Tetris V2 현장 패치 검증 리포트

## 1. 기준

- 작성일: 2026-06-13
- 브랜치: `v2`
- 제품 구현 검증 기준 커밋: `5459d36`
- 목적: 2026-06-12 현장 패치 범위가 자동 검증과 대표 현장 케이스 기준을 만족하는지 추적한다.

이 리포트는 `docs/plans/2026-06-12-v2-field-patch-plan.md`의 Phase 6 산출물이다. V2는 프론트 단독 구조를 유지하며, 검증된 변경만 `v2`에 push하고 안정화 후 `main`으로 병합한다. 이 리포트는 최신 HEAD를 자동 보증하지 않는다. 새 제품 구현 후에는 실제 터미널 출력 기준으로 기준 커밋, 테스트 수, 검증 결과를 함께 갱신한다.

검증 메타데이터 기준 파일은 `docs/verification/2026-06-13-v2-field-patch-verification.meta.json`이다. 문서 테스트는 이 파일의 커밋과 테스트 수가 리포트 본문에 반영되어 있는지 확인한다.

## 2. 역할별 최종 검토 메모

### Product Manager

- V2 패치의 핵심 위험은 계산 정합성과 현장 UI 해석성이다.
- 이번 리포트는 구현 완료 선언이 아니라, 다음 사이클에서도 같은 기준으로 재검증할 수 있게 남기는 기준 문서다.

### Business Analyst

- 현장 사용자는 숫자 결과와 3D 화면을 함께 보고 오버 여부를 판단한다.
- `결과 최대치수`는 현재 결과가 실제로 차지한 가로/세로/높이 끝 좌표로 설명해야 한다.

### UI Designer

- 3D 치수 표시는 캔버스 하단을 덮지 않는 compact chip bar를 기준으로 유지한다.
- 사용자 문구는 `가로/세로/높이`를 사용하고 `깊이` 표현은 새 안내에 남기지 않는다.

### UI/UX Tester

- UI 변경이 있는 사이클은 360px, 390px, 768px, 1280px에서 horizontal overflow와 주요 CTA 위치를 확인한다.
- 3D 캔버스의 `결과 최대치수` 오버레이는 조작을 막지 않아야 한다.

### Code Reviewer

- 엔진 관련 완료 조건은 단일 성공 예시가 아니라 boundary, collision, support, fragile policy, determinism, utilization 검증을 함께 통과하는 것이다.
- field audit summary에서 바람개비 세부 항목이 빠지면 엔진 정합성 패치를 완료로 보지 않는다.

### Next.js Developer

- 서버 기능은 추가하지 않고, Next.js static export와 IndexedDB/JSON 백업 구조를 유지한다.
- 새 검증 기준은 문서 테스트로도 고정해 다음 사이클에서 누락을 방지한다.

## 3. 자동 검증 결과

| 명령 | 기대 결과 | 기록 |
| --- | --- | --- |
| `npx next typegen` | Next.js 자동 생성 타입 파일 준비 | 통과 |
| `npm test` | Node test runner 전체 통과 | 439개 테스트 통과 |
| `npx tsc --noEmit` | TypeScript 타입 검사 통과 | 통과 |
| `npm run field:audit` | 현장 preset과 V2 기능 검증 통과 | `Package Tetris 현장 audit 통과` |
| `npm run build` | Next.js production build 성공 | `Compiled successfully` |
| `git diff --check` | 공백 오류 없음 | 통과 |

위 명령은 각 구현 사이클 종료 전 새로 실행해야 한다. 이 문서는 기준 항목을 남기는 리포트이며, 최종 커밋 전에는 실제 터미널 출력으로 다시 확인한다. 문서 테스트는 오래된 커밋과 테스트 수가 그대로 정답으로 남지 않도록 이 섹션을 함께 검증한다.

## 4. 대표 현장 케이스 재현 로그

### 4.1 바람개비 배치 기본 8개

- 입력 공간: `기본 파레트`, 1100 x 1100 x 1550mm
- 입력 박스: 690 x 370 x 580mm, 8개
- 검증 이름: `현장 바람개비 적재 검증 - 기본 8개`
- 기대 결과: 1공간, 적재 8개, 미적재 0개
- 완료 기준: 경계 초과 없음, 충돌 없음, 공중 배치 없음, 지지면 검증 통과

### 4.2 치수 순서 변형

- 검증 이름: `현장 바람개비 적재 검증 - 치수 순서 변형`
- 대상: `370 x 690 x 580mm`, `580 x 370 x 690mm`, `690 x 580 x 370mm`
- 기대 결과: 각 변형 케이스가 같은 안전 검증을 통과한다.

### 4.3 9개 경계

- 검증 이름: `현장 바람개비 적재 검증 - 9개 경계`
- 입력 박스: 690 x 370 x 580mm, 9개
- 기대 결과: 1공간에 무리하게 9개를 넣지 않는다. 2공간 또는 미적재 처리여도 안전 검증을 통과해야 한다.

### 4.4 주변 치수

- 검증 이름: `현장 바람개비 적재 검증 - 주변 치수`
- 대상: `691 x 370 x 580mm`, `690 x 371 x 580mm`, `690 x 370 x 581mm`
- 기대 결과: 바람개비 기본 케이스의 좌표를 그대로 복사하지 않고, 결과 전체가 안전 검증을 통과한다.

### 4.5 혼합 추가 시뮬레이션

- 검증 이름: `현장 바람개비 적재 검증 - 혼합 추가 시뮬레이션 결과`
- 기준 입력:
  - 200 x 150 x 200mm, 60개
  - 1000 x 800 x 400mm, 5개
  - 965 x 300 x 200mm, 10개
  - 600 x 250 x 150mm, 10개
- 추가 시뮬레이션:
  - 1000 x 800 x 400mm, 1순위, 6개 지정
  - 965 x 300 x 200mm, 2순위, 최대
  - 600 x 250 x 150mm, 3순위, 최대
- 완료 기준: 추천 결과, 선택 순서 결과, 각 박스 우선 variant가 모두 안전 검증을 통과한다.

## 5. UI 검증 기준

- `결과 최대치수`는 결과 3D 안에서 가로/세로/높이 순서로 표시한다.
- `깊이` 표현은 새 UI와 문서 안내에 사용하지 않는다.
- 3D 치수 오버레이는 compact chip bar로 표시하고, 캔버스 조작을 막지 않도록 `pointer-events: none`을 유지한다.
- 브라우저 검증 폭은 360px, 390px, 768px, 1280px을 기준으로 한다.
- 결과 3D는 빈 캔버스가 아니어야 하며, `3D`, `위`, `앞`, `옆`, `처음`, `크게 보기` 버튼이 유지되어야 한다.
